# Query Explanations - Day 7 SQL Detective Day

## Fixed Queries Explanations

## Query 1 - Count orders by city

### What the query does:

This query counts how many orders exist in each city. It groups orders based on the customer's city and calculates the number of orders for every location.

### Tables used:

* `orders`
* `customers`

### Why JOIN is needed:

A JOIN is needed because the city information is stored in the `customers` table, while order information is stored in the `orders` table. The relationship between them is created using `customer_id`.

### What the result means:

The result shows which cities generate the most orders. This helps the business understand where customers are located and where sales activity is strongest.

---

## Query 2 - Calculate revenue by product

### What the query does:

This query calculates revenue generated by each product by multiplying the ordered quantity by the product price.

### Tables used:

* `orders`
* `products`

### Why JOIN is needed:

A JOIN is required because the product name and price are stored in the `products` table, while quantity and order status are stored in the `orders` table.

### What the result means:

The result identifies which products generate the most revenue and helps the business decide which products are most important.

---

## Query 3 - Show completed orders only

### What the query does:

This query filters the orders table and returns only orders with a status of `completed`.

### Tables used:

* `orders`

### Why JOIN is needed:

A JOIN is not needed because all required information exists inside the orders table.

### What the result means:

The result shows successful purchases that can be counted as real business transactions.

---

## Query 4 - Find cancelled orders

### What the query does:

This query finds all orders where the order status is cancelled.

### Tables used:

* `orders`

### Why JOIN is needed:

A JOIN is not needed because the status column exists directly in the orders table.

### What the result means:

The result shows orders that did not generate revenue and may require investigation to understand why they were cancelled.

---

## Query 5 - Revenue by customer

### What the query does:

This query calculates the total completed revenue generated by each customer.

### Tables used:

* `orders`
* `customers`
* `products`

### Why JOIN is needed:

JOINs are needed because customer names come from the customers table and product prices come from the products table. The orders table connects these tables together.

### What the result means:

The result shows the highest-value customers and helps identify loyal customers.

---

## Query 6 - Revenue by category

### What the query does:

This query groups completed sales by product category and calculates total revenue for each category.

### Tables used:

* `orders`
* `products`

### Why JOIN is needed:

The category information exists in the products table, but the purchased quantity exists in the orders table, so both tables are required.

### What the result means:

The result shows which categories contribute the most money to the business.

---

## Query 7 - Find customers with multiple orders

### What the query does:

This query counts customer orders and returns customers who have placed more than one order.

### Tables used:

* `orders`
* `customers`

### Why JOIN is needed:

## Query 8 - Orders that should not count as revenue

### What the query does:

This query finds orders that have a status such as pending or cancelled.

### Tables used:

* `orders`

### Why JOIN is needed:

### What the result means:

The result prevents incorrect revenue calculations by showing transactions that should be excluded.
